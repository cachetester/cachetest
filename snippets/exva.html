Explicit validation of responses without freshness lifetime The results in this table depict if the caches perform a freshness validation for equivalent requests when origin server force them to do it with the control directive no-cache. A freshness validation, also known as conditional request, is performed based on validation tokens included in the stored response. RFC 7234 defines two types of validation tokens: (1) an opaque entity tag represented by the ETag header field and (2) a time-variant parameter that can be retrieved form the Date or Last-Modified header. If a response does not comprise the ETag, Last-Modified or Date header, a conditional request may contain a self-defined timestamp derived from a cache-internal clock. A freshness validation request intending to verify with an opaque token must contain the If-None-Match header comprising one or multiple entity tags. Freshness validation requests based on a time-variant token require to append the If-Modified-Since header including either a self-defined timestamp or the timestamp value of the Date or Last-Modified header. A successful freshness validation requests results in a response with a 304 Not Modified status code. The symbol [ok] indicates that the cache transform equivalent requests into conditional requests. The symbol [fail] indicates that a equivalent request is not transformed into conditional requests, meaning that the cache send a request without any condition to retrieve the full response with body. If caches perform an explicit conditional request but does something wrong after the validation, we mark this with ([ok]). More details on the issues can be see at the info icon.
